## 0. What this project is doing

We're extending Swiss AI's **Apertus-8B-2509** tokenizer for Greek by
training on a cleaned + deduplicated GlossAPI corpus (+ an HPLT Greek
slice). Four tokenizer arms are produced per wave:

| arm | mix | strategy |
|---|---|---|
| F1 | glossapi-only | fresh BPE discovery (50k vocab) |
| F2 | glossapi + 70/30 HPLT | fresh BPE discovery (50k vocab) |
| C1 | glossapi-only | continuous BPE from Apertus (target 156,672 vocab = base 131,072 + 25,600 extension) |
| C2 | glossapi + 70/30 HPLT | continuous BPE from Apertus |

We're in **wave 2** of the iteration. Wave 1 shipped earlier with the
old corpus.clean cleaner; wave 2 is the first pass with the unified
"Pilot B" parser-validated cleaner + per-rule counters from the
2026-04-25 cleanup plan. We discover residual noise in the trained
vocab, fix the cleaner, re-clean, re-train.

## 5. The iteration loop

The whole point of this subproject is the loop:

```
   ┌──────────────────────┐
   │   Cleaner update      │  Pilot B / Rule B / per-char filters
   │   (eellak/glossAPI)   │
   └─────────┬────────────┘
             │
             ▼
   ┌──────────────────────┐
   │   Re-clean canonical  │  reclean_canonical_to_parquet.py
   │   (gcloud, full corpus)│  → glossapi_work_wave2_<date>/canonical/
   └─────────┬────────────┘
             │
             ▼
   ┌──────────────────────┐
   │   Tokenizer training  │  wave2_orchestrate.py: dedup → mix → train ×4
   └─────────┬────────────┘
             │
             ▼
   ┌──────────────────────┐
   │   Tokenizer analysis  │  pull tokenizer.json → inspection/
   │                       │  extract greek / non-greek / special-char
   │                       │  audit; spot leftover noise families
   └─────────┬────────────┘
             │
             ▼
   ┌──────────────────────┐
   │   Decide rules         │  classify each finding:
   │                       │   - normalize (rewrite to canonical form)
   │                       │   - match+strip (drop noise)
   │                       │   - leave (legitimate content)
   └─────────┬────────────┘
             │
             └─→ next iteration: update cleaner with the rules → loop
```

Each iteration is a **wave**; wave 1 was the baseline corpus.clean
cleaner, wave 2 is the unified Pilot B cleaner, wave 3 will land the
new normalize / strip rules surfaced in this iteration.

### How to investigate the latest tokenizer for residual noise

The standardised check, using the four extraction artifacts in the
inspection dir:

1. **Top of the vocab.** Sort `greek_tokens.jsonl` by `id` ascending.
   The first ~5,000 ids are the most frequent BPE merges. Any junk
   here is corpus-frequent and must be cleaned — but most of it
   should be normal Greek subwords.
2. **Special-char buckets.** Read `special_token_audit.md` (auto-gen
   from `audit_special_tokens.py`). It buckets letter-and-digit-free
   tokens by category (run-of-`-`, run-of-`_`, mixed-punct, control
   chars, etc.). Any bucket with non-trivial counts is a candidate for
   a normalize / strip rule.
3. **Non-Greek list.** `non_greek_tokens.jsonl` (13,175 tokens for F1).
   Filter by patterns the audit doesn't catch — things like
   `GLYPHGLYPH`, `/hyphenminus`, `ïõ` (mojibake), single-char
   tokens for mojibake bytes (`Î`, `Ï`, `ï`, `â`).
4. **Cross-arm comparison** (when wave-2 has all 4 tokenizers):
   compare F1 / F2 / C1 / C2 vocabs side-by-side; tokens that appear
   in all four are the most robust candidates for normalize rules.
5. **Empirical examples.** Pull doc samples that produce a suspect
   token; trace where in the cleaner pipeline the residue survives.
   Don't claim a "root cause" without this trace step (we got burned
   on the GLYPH bug — see "Implementation plan" section in
   `corpus_clean_normalization/NORMALIZATION_DESIGN_20260420.md`).

## 6. The wave-2 audit findings (what's open right now)

Detailed in **`corpus_clean_normalization/NORMALIZATION_DESIGN_20260420.md`**
under the section **"Wave-2 audit findings (2026-04-27)"**. That doc
is the source of truth; this section is a 30-second summary.

The wave-2 F1/F2 vocab comparison plus the independent F1 train-corpus
scan narrowed the wave-3 scope. Full details are in
`subprojects/01_0_cleaning_iteration_and_thresholds/WAVE3_CLEANER_PATCH_PLAN_2026-04-28.md`.
Findings still split into NORMALIZE (rewrite content) and MATCH+STRIP
(drop noise), but the active list is smaller than the first audit.

**NORMALIZE** (active for wave-3):
- **A.** Continuous-run quantization to {1,3,5,20} (snap-DOWN floor)
  — extends existing `bucket_run_length` / `normalize_char_runs_tiered`
  to dash, asterisk, equal, hash, tilde, slash, backslash, pipe,
  exclamation, percent, at, caret, standalone tonos. Decision:
  length 4 maps to 3, not 5.
- **A.1** Markdown-escape unescape pre-pass (`\_\_…` → `__…` →
  quantize), generalized to escaped run characters such as `\*`,
  `\-`, `\#`, `\.`, `\=`, and `\~`.
- **A.2** Markdown table separator quantization (per-cell `-` runs).
- **B.** Combined `…` + `.` runs unified (`…` = 3 dots, then quantize).
- **C.** ATX heading 7+ → quantize; setext underlines → quantize.
- **E.** µ → μ — already in `fold_codepoint`; only small verification
  remains, likely around fenced-code pass-through.
- **G.** Repeated standalone tonos — covered by A.

**NORMALIZE — DEFERRED:**
- **D.** Mojibake-encoded Greek bigrams (`ïõ` etc.) — needs wave-3
  multilingual false-positive testing before landing.
- **F.** Cyrillic / Latin homoglyph fold — enough signal to
  investigate, but too easy to over-correct without a calibrated sample
  set. Tracked with D in <https://github.com/eellak/glossAPI/issues/99>.

**MATCH AND STRIP** (token-level only — line-level deferred):
- **H.** C0/C1 + soft hyphen — `is_unicode_noise_char` already covers,
  **needs verification** why these survived to F1 vocab (likely
  code-fence pass-through at `cleaning_module.rs:700-707`).
- **I.** Bare `GLYPH`+ residue — extend existing glyph Rule A/B logic
  in `apply_glyph_span_strip_and_rule_b`, preserving its count /
  coverage line-drop threshold.
- **M.** PostScript / PDF bare glyph-name residue (`/hyphenminus`,
  `/elipsis`, `/period`, `/comma`, `/space`, `/colon`) — same glyph
  Rule A/B extension, narrowed to high-confidence glyph contexts. Do
  **not** land generic single-letter glyph names, `/pi`, slash
  acronyms, units, or URL/document paths in wave 3.

**MATCH AND STRIP — DEFERRED (line-level):**
- **J.** Empty `$$ $$` math-fence residue.
- **K.** Standalone `Page \d+` page-marker residue.

**KEEP / NOT CLEANUP TARGETS:**
- **L.** Pictograph / dingbat tokens — user decision: keep them.
- Intentional HTML comment placeholders (`<!-- image -->`,
  `<!-- text-missing -->`, `<!-- formula-not-decoded -->`) — keep them
  and exclude them from bad-token cleanup accounting.

## 6.5 Parallelization gaps

The wave-2 production run surfaced several pipeline stages that are
single-threaded by design and waste the 64-vCPU instance. Three
families:

### 6.5.1 Dedup library (most-detailed; documented elsewhere)

Full breakdown with file:line refs in
`subprojects/01_1_corpus_dedup/DEDUP_SCRIPT_REPAIR_PLAN.md`, section
**"Wave-2 empirical findings (2026-04-26 → 2026-04-27, 49.3M docs)"**.
Ten entries, ordered by ROI:

1. `_run_near_cluster_stage` per-row SQLite INSERT loop — **biggest pathology**, ~1.5h single-thread for cosmetic summary stats. (`text_dedup.py:6236-6280`)
2. `build_stage_results` strict + relaxed exact rebuild — was projected 4-8h serial; worked around in wave-2 via `subprojects/01_1_corpus_dedup/scripts/parallel_exact_stages.py` (32s in DuckDB). (`text_dedup.py:2243-2415`)
3. Skip-if-exists guards for `write_run_docs_inventory` (24 min single-thread, `text_dedup.py:1769`) and `write_snapshot_manifest` (10 min, `text_dedup.py:2418-2440`).
4. Per-shard + consolidated double storage — 290 GB intermediate waste (Stage 2 shards never cleaned after `combine_parquet_files`).
5. `COPY TO parquet` directly instead of fetch-record-batch + ParquetWriter for `write_docs_exact_export` and `_build_final_exports` — DuckDB query is parallel, but the consumer is single-thread.
6. Disk preflight at Stage 2 start (1.58B LSH bucket entries × per-band parquet ≈ 700 GB; we hit errno 28 mid-run).

The wave-2 workaround (parallel_exact_stages.py) is the proof-of-
concept for fix #2; #1 is the next-biggest single lever.

### 6.5.2 Mix command post-write verification — UNDOCUMENTED

Observed in wave-2 production:
- `mix_glossapi_only` wrote `mix.parquet` (17.7 GB) at 15:17 UTC, then spent ~45 min reading it back at ~5 cores busy (load 4.5–7) before the orchestrator advanced.
- `mix_glossapi_plus_hplt_70_30` similar — wrote 30 GB at 17:01, then ~1h+ verification before advancing.

That's roughly **1.5h of largely-serial post-write work per mix arm**.
Likely the same DuckDB-query → single-thread-ParquetWriter anti-pattern
as the dedup `_build_final_exports` case.

**Trace target:** `glossapi_corpus_cli/pipeline.py`, the post-write
phase of `build_mix_output_from_selected_input` (or whichever function
emits `mix.parquet` and computes `mix.summary.csv` /
`mix.source_mix_summary.json`). Search for `fetch_record_batch` and
`ParquetWriter`; replace with `COPY (… ) TO 'mix.parquet' (FORMAT
parquet, COMPRESSION zstd)` style.

### 6.5.3 Continuous BPE training `count_segments` phase — UNDOCUMENTED

The arm we killed at 22:21 UTC on 2026-04-27. Single-thread BPE-encode
+ frequency-count over the 16.5 GB train.parquet. **2h35m+ with no
phase advance.** CPU 99.9% on one core; the ~10 multiprocessing
children spawned by the script were sitting at 0% CPU (work not
dispatched to them).

Source: `subprojects/02_1_tokenizer_experiments/scripts/train_continuous_bpe_tokenizer.py`,
or its imports from `glossapi_corpus_cli.continuous_bpe`
(`build_sequence_shards`, `aggregate_sequence_shards`,
`collect_segment_shards`).

**Trace target:** the script imports `build_sequence_shards` which
suggests parallel-able shard primitives; the question is why the
parent process serializes the work instead of dispatching shards to
the pool. Likely a `for shard in shards: do_one_shard(shard)` somewhere
that should be `pool.map(do_one_shard, shards)`.

### 6.5.4 `train_discovery_tokenizer.py` (F1, F2) — VERIFY

F1 took 38 min on 16.5 GB train, F2 same on 30 GB train (so it
parallelizes at least somewhat — F2 is 1.8× the data and finished in
the same wall time). Probably internally OK because HuggingFace
`tokenizers.train_from_iterator` uses Rayon under the hood. **Verify
via `top` during the next training run** — if all 64 cores are warm,
no work to do; if only a few are warm, the iterator wrapper is the
bottleneck and we should refactor to feed it batched parallel shards.

### 6.5.5 Cleaner (`reclean_canonical_to_parquet.py`) — already parallel

Phase 1 re-clean ran 49.3M docs in 67 min on 64 ProcessPool workers.
Confirmed parallel; not a target. (One known follow-up — submission-
order parent-side harvesting; doesn't block, just delays the
`reclean_summary.jsonl` view.)

### Order of work (by impact)

1. **Mix post-write verification (6.5.2)** — saves ~1.5h × 2 arms = 3h per wave.
2. **Continuous BPE `count_segments` (6.5.3)** — saves multiple hours per C-arm × 2 arms.
3. **Dedup `run_near_results` INSERT loop (6.5.1 #1)** — saves ~1.5h per dedup run.
4. **Dedup `build_stage_results` rebuild (6.5.1 #2)** — code review of `parallel_exact_stages.py` and lift it into the library.
5. **Disk preflight + auto-cleanup of consolidated stages (6.5.1 #4, #6)** — prevents disk-full crashes that derailed wave-2.

Sum of items 1-3: ~6 hours per wave, on a $7-8/hr instance — **~$50 saved per wave** if all three are landed.
