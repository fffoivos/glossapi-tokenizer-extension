# CSCS Auth + Execution Workflow

*Rewritten 2026-05-20 against the **new `cscs-key` Rust tool** (v1.1.0,
released April 2026, github.com/eth-cscs/cscs-key). The old
`sshservice-cli/cscs-keygen.{sh,py}` tool is deprecated.*

## What changed

| | Old (sshservice-cli, deprecated) | New (`cscs-key` v1.1.0+) |
|---|---|---|
| Tool | `cscs-keygen.sh` / `.py` (bash + python) | `cscs-key` (Rust static binary) |
| Auth | username + password + 6-digit OTP, POSTed to `sshservice.cscs.ch/api/v1/auth/ssh-keys/signed-key` | OIDC browser login (or device-code flow with `--headless`) |
| First-time key | tool generated both private and public on the server | you generate the local pair yourself with `ssh-keygen -t ed25519 -f ~/.ssh/cscs-key`, the tool only **signs** the public half |
| Daily | rerun keygen, prompt for OTP each time | `cscs-key sign` — OIDC token is cached locally, no daily MFA prompt |
| Subcommands | only key generation | `sign`, `list`, `revoke`, `completion`, `gen` (hidden, deprecated) |

The bigger conceptual shift: under the old tool, CSCS minted your
private key for you (server-side). Under the new tool, **your private
key lives only on this machine** — CSCS only ever signs the public
half. That's a cleaner trust model. The flip-side is that you have to
explicitly create the keypair the first time.

## One-time setup on `home`

### 1. Install `cscs-key`

Linux home is x86_64 → use the musl static binary so it works on any
glibc version:

```bash
mkdir -p ~/.local/bin
cd /tmp
curl -L -O https://github.com/eth-cscs/cscs-key/releases/download/v1.1.0/cscs-key-v1.1.0-x86_64-unknown-linux-musl.tar.gz
tar -xzf cscs-key-v1.1.0-x86_64-unknown-linux-musl.tar.gz
# tar contains `cscs-key` binary directly
mv cscs-key ~/.local/bin/
chmod +x ~/.local/bin/cscs-key
cscs-key --version    # sanity check
```

(`~/.local/bin` is already on `PATH` for this user's bash.)

### 2. Create the local SSH keypair (only once)

The existing `~/.ssh/cscs-key` was generated by the old server-side
flow. Under the new tool, you generate it locally:

```bash
# Back up the old key + cert in case of trouble (don't overwrite blindly)
mv ~/.ssh/cscs-key      ~/.ssh/cscs-key.old.$(date +%Y%m%d) 2>/dev/null
mv ~/.ssh/cscs-key.pub  ~/.ssh/cscs-key.pub.old.$(date +%Y%m%d) 2>/dev/null
mv ~/.ssh/cscs-key-cert.pub ~/.ssh/cscs-key-cert.pub.old.$(date +%Y%m%d) 2>/dev/null

# Generate the new pair
ssh-keygen -t ed25519 -f ~/.ssh/cscs-key -N "" -C "fffoivos@home $(date +%Y-%m-%d)"
# -N "" leaves the key passphrase-less for Claude-driven workflows;
# add a passphrase here if you'd rather gate every use behind it.
```

After this step:
- `~/.ssh/cscs-key` — private (chmod 600)
- `~/.ssh/cscs-key.pub` — public, **unsigned** (chmod 644)

The signed certificate `~/.ssh/cscs-key-cert.pub` doesn't exist yet —
that comes from `cscs-key sign` (next step).

### 3. Existing `~/.ssh/config` aliases still apply

The `ela`, `clariden`, `bristen` Host blocks point at
`IdentityFile ~/.ssh/cscs-key` + `CertificateFile ~/.ssh/cscs-key-cert.pub`.
Those still work — only the way the cert is *minted* has changed.

## Daily workflow

### You do:

```bash
cscs-key sign
```

Default validity is **1 day**.

> **Note (verified 2026-05-20 against `cscs-key 1.1.0`):** the only
> accepted `--duration` values are `1d` and `1min`. CSCS policy caps
> certs at 24 h via this tool; the GitHub release notes' mention of
> custom durations does not apply to v1.1.0. If you want a longer
> validity, you need to either (a) re-run `cscs-key sign` each day,
> or (b) chain it from cron / systemd-timer with a stored OIDC
> refresh token. For now: **re-run daily**.

What happens:

1. The tool opens a browser tab pointing at the CSCS OIDC login page.
2. You authenticate with CSCS credentials (whichever flow CSCS has bound — typically EduID + MFA).
3. The signed cert is written to `~/.ssh/cscs-key-cert.pub` (chmod 644).
4. An OIDC refresh token is cached locally so subsequent `cscs-key sign` calls don't reprompt for ~24 h.

#### When the browser is not available on `home`

`home` is a headless server reached over SSH from your laptop. In that
case use the device-code flow:

```bash
cscs-key sign --headless
```

That prints a URL + 8-char device code; you visit the URL on any
browser, paste the code, approve. The cert lands on `home` once
approved. This is the right mode for the SSH-to-home scenario.

### Then tell me "cert refreshed"

I run this health check before any cluster operation:

```bash
ssh-keygen -L -f ~/.ssh/cscs-key-cert.pub 2>/dev/null | grep -E 'Valid:|Key ID:|Principals:'
# or, equivalently, using the new tool:
cscs-key list
```

`cscs-key list` shows your active certs with serial, expiry, and key ID.
If the cert in `~/.ssh/cscs-key-cert.pub` is already expired or expires
within the next 30 minutes during a long-running session, I stop and
ask you to refresh.

## Working launch pattern (already verified)

From `/home/foivos/Projects/cscs-clariden-project-understanding/`,
the verified-working multi-node DDP invocation is:

```bash
ssh clariden
cd /iopsstor/scratch/cscs/fffoivos/clariden-toy-pytorch
./run_distributed_2node.sh a0140
# expands to:
#   srun -A a0140 -p normal -N 2 --ntasks-per-node=1 --time=00:08:00 ./node_entrypoint.sh
# node_entrypoint.sh runs:
#   uenv run pytorch/v2.6.0:v1 --view=default -- torchrun --nnodes=2 --nproc_per_node=4 ...
```

The `--view=default` flag is non-obvious but **required** — without it
`python3` falls back to a base interpreter and the modern Torch stack
isn't on PATH.

## Open items before first real run

1. **Confirm we have `cscs-key` installed locally.** Once you've run the install steps in §"One-time setup" the first time, this is a no-op going forward.
2. **First `cscs-key sign` of the new flow** — picks an OIDC method (EduID? CSCS user-account?). The first sign may surface a different identity provider than the old tool used.
3. **Verify ssh still works** via `ssh ela 'hostname'` after the first new-flow cert is minted. If the cert principal changed from `fffoivos` to something else, that's an `~/.ssh/config` update.
4. **Once auth is working**: pick partition + node count for the three-arm pilot. The Clariden node shape is 4× GH200 120 GB, 288 vCPU, ~856 GiB RAM. For a 10 B-token Apertus-8B pilot, 1 node (4 GPUs) should give ~50-80k tokens/s with FlashAttention-2; 2 nodes (8 GPUs) ~100-160k tokens/s. Pilot of 10 B tokens on 1 node ≈ 1–2 days; on 2 nodes ≈ 18-24 h.
