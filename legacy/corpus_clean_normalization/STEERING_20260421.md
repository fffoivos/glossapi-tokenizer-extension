## Corrections, in rough chronological order

### C1. Prompts didn't match the designed structure

**User**: "the prompts are still not like the structure I asked you …
they don't read fluently, don't preserve the line breaks etc of the
original text."

**What happened**: wave11 case files were generated in `char window
fallback` mode. The designed 20-line context windows didn't
materialize because my `_build_v2_case_body` called `Path(source_path).is_file()`
and `source_path` was actually `<parquet>#<doc_id>` (matcher ran over
parquet rows, not markdown files). Fallback returned 240-char slices
instead of 20-line windows. Legacy review modes (`cleaning`,
`normalization`) used the wave10 template entirely, with no `[MATCH]`
tag and no shadow-normalized view.

**Going forward**: if we rerun wave11, the bundler must resolve
`<parquet>#<doc_id>` into real row text via `pd.read_parquet(...)` with
a per-parquet cache, and apply the v2 renderer to ALL review modes
(not just the new ones). No "fallback mode" output shipped to Gemini.

### C2. Match tags are too verbose

**User**: "match tags are too long."

**What happened**: `<match type="control_private_use_replacement"
family="fresh_glossapi_only_control_private_use_replacement">`. The
`family` attribute is an internal matcher identifier with no signal
for the model.

**Going forward**: compact per-category tag forms:
- Category-only: `<match kind="pua/control">…</match>`
- When the raw trigger is a single codepoint:
  `<match trigger="U+200E">…</match>` — surfaces the actual problem
  directly instead of dragging context.

### C3. The BPE-literal-set matcher drags context

**User**: "we dont want to remove the whole match though just the
invisible character. so why dont we look just for that?"

**What happened**: the matcher's `control_private_use_replacement`
(and similarly `table_border_ascii_art`) categories use a
`literal_set` of 391 multi-character BPE tokens — each containing at
least one problem codepoint alongside whatever neighbor chars the BPE
merged. Result: the matched anchor shown to reviewers is e.g.
`: Zwis` (with an invisible U+200E at the end) instead of just the
U+200E itself, or a full Greek academic sentence because the sentence
happened to end with `[`.

**Going forward**: for codepoint-level noise, match the codepoint
directly (regex / small literal set of single chars), not the full
BPE tokens. Matched anchor = actual problem. Deterministic rather
than routed to Gemini.

### C4. Invisible chars don't need Gemini review — strip directly

**User**: "I thought we had already agreed that these invisible chars
dont need any gemini review…"

**What happened**: after the matcher-specs insight (C3) I wrote a
per-codepoint specs file routing U+200E to a codepoint-based category,
still pointed at Gemini review.

**Going forward**: if a codepoint is invisible AND has no semantic role,
it's unambiguously noise → direct strip via `is_unicode_noise_char` in
the cleaner, **no review at all**. Gemini time is for ambiguous
cases where the boundary is genuinely context-dependent (e.g.,
"is this span a separator or a statistical marker?"), not for
confirming the obvious. The fix for C4 was a two-codepoint addition
(U+200E, U+200F) to `is_unicode_noise_char`, not a matcher/review
pipeline.

### C6. Evidence before stripping — corpus-specific

**User**: "we only want to remove them if there is indication that
they are the noise. Our corpus language is meant to be greek, so of
course there will be ancient, modern + academic/legal content with
math, other languages especially european, etc."

**What happened**: commit `9ea674f` added Armenian/Hebrew/Arabic/
Georgian ranges to `unusual` because they're "non-Greek-non-Latin
scripts likely to be mojibake." I didn't check the bad-token
inventory first. Armenian + Hebrew have **zero** tokens in the
14,182-token inventory; Arabic + Georgian have 7 + 8 trace tokens.
Not enough to flag as noise; risk of stripping legitimate
quotations in humanities work.

**Going forward (policy)**: before adding any codepoint range to a
strip list, answer three questions empirically:
1. Is it in Apertus's vocab as a single-token merge?
2. How many times does it appear in the fresh-BPE bad-token inventory?
3. On noisy pages in the corpus, does it dominate, or is it rare /
   scattered?

The cost/benefit is asymmetric: over-cleaning legitimate foreign
quotations is worse than letting rare scripts pass through.

The existing `corpus.clean` `unusual` set already made these calls
carefully (Cyrillic stripped — Greek homoglyph concern; Armenian et
al. preserved — legitimate quotation concern). Don't disturb it
without evidence.

### C7. Math-Greek: check Apertus first, then fold to plain Greek

**User**: "we should be checking if something is in the Apertus
tokenizer before doing that. (2) if it isn't we can decide into
folding/keeping those that have semantic meaning, we only strip
those that have no semantic meaning i.e. noise!"

**What happened**: I put Math Alphanumeric Greek (U+1D6A8..U+1D7C9)
into `unusual` (strip), assuming Apertus doesn't have them and
they're noise.

**Evidence check (done after user pushed back)**: every Math-
Alphanumeric-Greek codepoint tokenizes as **4 tokens** in Apertus
(raw UTF-8 bytes); plain Greek is **1 token**. Math-Greek is not in
Apertus's merges, but it has semantic meaning (Greek letter
identity) — it's a presentation variant of a real letter. Correct
action: **fold to plain Greek** (4 tokens → 1 token, preserves
meaning). Shipped in commit `7e9db6d`.

**Going forward (policy)**: for any codepoint not in Apertus's merges,
decide:
- Has semantic meaning? → fold to a codepoint Apertus does tokenize.
- No semantic meaning? → strip.
- Ambiguous? → measure in the corpus first.

### C8. Whitespace has semantic roles beyond prose — TOC leaders, indent

**User** (first time): "whitespaces are also used in TOC sometimes
instead of dots so we need a standardized length like with dots,
this should be different than mapping 3 spaces to 1."

**User** (second time, after I wrote a TOC-specific rule): "Let's
say >10 gets normalized to something similar to the dots i.e. 20
whitespaces. And <=10 to 1 or whatever else is reasonable. This
should be uniform to the dots. Eg for {1,2} -> 1, {4,10} -> 3, >10
-> 20."

**User** (third time, after I asked about failure modes): "we can aim
for tokens of . ... and ..... so that we collapse all to the nearest
divisible by 5 or something. unless >20 ->20."

**What happened**: I started with a generic "2+ spaces → 1 space"
collapse. User flagged this destroys the TOC signal. I wrote a
narrow TOC-specific heuristic. User reframed: don't have a special
case, just apply a uniform tiered bucket. Final target vocabulary
`{1, 3, 5, 20}` uniform for dots and whitespace. Shipped in commit
`f4f4314`.

**Going forward (policy)**:
- Don't special-case patterns when a uniform tiered rule captures
  them.
- Before collapsing any layout chars, ask: what role does this
  pattern play in real docs? Long whitespace runs carry TOC / column
  alignment info, not just noise.
- Visible target tokens (e.g., 20 spaces) preserve the reader's
  perception of the original structure where a 1-space collapse
  destroys it.

### C9. Don't run Gemini on decisions the parser makes

**User**: "why are you matching md table lines? I thought you had
developed a deterministic method for this."

**What happened**: wave11 submitted 300 `markdown_table_separator_like`
cases to Gemini for per-case review in a mode called `md_table_audit`.
This ran in parallel with the deterministic GFM parser in the cleaner

**Going forward (policy)**: if a decision is made by a deterministic

**Ahead on current branch `codex/token-noise-review-debug`, NOT yet
merged to `development`** (`rust/glossapi_rs_cleaner/src/normalize.rs`
— added here):

- Tiered bucket `bucket_run_length(n)` → `{0|1→n, 2→1, 3→3, 4..=20→5, _→20}`.
- `normalize_dot_runs` — dot runs bucketed (TOC leaders land on
  `.....` / `....................`).
- `normalize_whitespace_runs` — whitespace runs bucketed; leading indent
  preserved so markdown indented code blocks survive.
- `normalize_separator_line` — standalone separator lines → canonical.
- Ellipsis-run + malformed-entity normalization.
- `fold_codepoint` — Math Alphanumeric Greek → regular Greek.
- `drop_low_salvage_pages` — page-salvage helper.

**No tests currently exist** for GFM-table canonicalization (e.g.
taking `| -------- | -------- |` → `| --- | --- |`). That is a pure
future item — implementation + tests both missing.

**In scope for future normalization**

1. **GFM markdown tables — deterministic canonicalization.**
   Models emit more delimiter chars than needed
   (`| -------- | ---- |` vs `| --- | --- |`). Reduce each separator
   row to the minimal GFM-valid form. Column count, cell contents, and
   visible layout intent stay unchanged. Parser-based
   (`markdown-it-py` / `pulldown-cmark`), not regex. Deterministic —
   direct rule, no review gate.

2. **Repeated special chars outside tables.** `\_\_\_\_\_\_…`,
   `______`, etc. Semantic role: creating visual space / distance /
   separation on the page for the reader. Bucket run lengths; don't
   strip them. Dots + whitespace are covered today. An **escape-aware**
   rule on `(?:\\_){N,}` is still missing — this is what EU legislative
   docs use for footnote-section dividers inside GFM table rows.

3. **Repeated special chars INSIDE markdown tables — careful, separate
   treatment.** Inside a table cell / separator row these still carry
   a structural role; a naive global rule can break the table. Design
   per-table-structure handling ("collapse runs but keep the cell
   valid and the column count intact") before implementing.

4. **Unified dot + ellipsis bucketing** (added 2026-04-21 — replaces
   the two separate `normalize_dot_runs` + `normalize_ellipsis_runs`
   rules). Each `…` (U+2026) is treated as exactly 3 dots in
   run-length counting; mixed `.` + `…` runs merge into a single
   run; output is ASCII dots (`...`, `.....`, `....................`)
   via the existing `bucket_run_length`. Target the
   `dot_ellipsis_mixed` BPE category (10 tokens: `….`, `…..`, `……..`,
   `……….`, `………..`, `…...`, …). Examples:
   - `….` = 4 dots → `.....`
   - `……` = 6 dots → `.....`
   - `……..` = 8 dots → `.....`
   - 32×`…` = 96 dots → `....................`
   - `…` alone = 3 dots → `...`
   Caveat: Apertus has `…` as a single-token vocab hit; reconstructing
   to ASCII may tokenize as 3 tokens in isolated cases. User preference
   stands: target is ASCII.

**FROZEN — do NOT match or rewrite unless a specific rule targets it**
- Generic markdown table lines — we do NOT match them for review or
  generic noise treatment. Every table-line touch must be justified by
  a specific table rule (e.g., item 1 above).
- LaTeX blocks / inline math — don't match, don't rewrite, unless a
  specific LaTeX-aware rule is in play.
- Code blocks / fenced code — already guarded; don't add generic
  matchers that reach into them.

### "Run" definition — locked (2026-04-21)

A **run** is a sequence of **≥ 2 identical chars** (length 1 is
passthrough, not a run). Every repeated-symbol normalization uses the
same bucket:

```
bucket_run_length(n) = {
    0 | 1  -> n,       // passthrough
    2      -> 1,
    3      -> 3,
    4..=20 -> 5,
    _      -> 20,
}
```

Applies uniformly to dot runs, whitespace runs, underscore runs,
escaped-underscore chains (future), hyphen runs inside tables
(future), ellipsis-unified dot runs (Feat-B), and any future
run-based rule. Any departure must be justified in a scope update.

### Code-segment guard — status (2026-04-21)

Currently guarded:
- **Fenced code blocks** (```` ``` ```` / `~~~`) — `core_clean_text`
  toggles `in_code_fence` at `cleaning_module.rs:300` and passes
  fenced lines through untouched (BAD_LINE_AC, tag strip, char filter,
  every normalize call — all skipped).
- `scan_gfm_table_separators` whole-doc pre-pass tracks its own
  independent fence state (`normalize.rs:524`).

Not guarded (open):
- **Indented code blocks** (4-space leading indent). Partial protection
  via `normalize_whitespace_runs` leading-indent preservation; other
  per-line rules still fire.
- **Inline code** (backticks inside a prose line). No guard today.

**Policy**: fenced-block guard is the invariant. Any new rule added to
`core_clean_text` or any whole-doc pre-pass MUST respect the
fenced-block state. Indented-code-block and inline-code guards are
open per-rule design questions.

### Decision table

|                 | direct (deterministic) | review-gated (Gemini / human) |
|-----------------|------------------------|-------------------------------|
| cleaning        | invisible-char strip, glyph-tag strip | ambiguous Math-block decisions |
| normalization   | GFM table canonicalization, run-length bucketing | uncertain separator-line cases |
| page-salvage    | drop pages > X% noise density | borderline pages needing human call |

If a proposed rule can't be placed on this grid, the scope is wrong —
rewrite it or drop it. This section is the strict contract.
